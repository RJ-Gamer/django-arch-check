> Goal: build a VS Code extension that shows the same `django-arch-check` HTML report as an interactive dashboard inside the editor, without re-implementing the analyzer or duplicating report logic.

---

## Vision

The extension should make `django-arch-check` feel native inside VS Code:

- Run analysis from the Command Palette
- Open the architecture report in a Webview panel
- Refresh the dashboard on demand
- Reuse the existing report HTML aesthetic and interactions
- Keep the Python package as the single source of truth for report generation

The extension should **not** create a second report engine in TypeScript. That would split logic across two codebases and make the CLI report and editor dashboard drift over time.

---

## Product Principles

### 1. Reuse Existing Report Output

The extension should consume the HTML already generated by `django-arch-check`, not rebuild the dashboard in React or plain TypeScript.

Why:

- the package already has a polished HTML report generator in `django_arch_check/report.py`
- the report already supports interactivity: filters, expand/collapse, theme switching
- keeping one renderer avoids double maintenance

### 2. Extension Owns Orchestration, Not Analysis

The extension should:

- discover the workspace root
- run the CLI
- capture the generated `arch-report.html`
- show it inside a Webview

The extension should not:

- parse Python source itself
- duplicate detector logic
- reimplement scoring logic

### 3. Keep the First Version Narrow

MVP should focus on:

- one dashboard panel
- one command to run analysis
- one command to refresh
- clean error handling

Inline diagnostics, Problems integration, SARIF parsing, and code actions can come later.

---

## Recommended Technical Strategy

## Core Decision

Use the Python CLI to generate `arch-report.html`, then load that HTML into a VS Code Webview.

This is the simplest path and the lowest-risk architecture.

### Preferred Flow

1. User runs `Django Arch Check: Open Dashboard`
2. Extension finds the active workspace folder
3. Extension runs:

```bash
django-arch-check analyze --format html <workspace-folder>
```

4. CLI generates `<workspace-folder>/arch-report.html`
5. Extension reads the generated HTML file
6. Extension injects it into a Webview panel
7. User interacts with the dashboard inside VS Code

### Why This Is Best

- zero duplication of report UI logic
- zero HTML-to-JSON-to-HTML translation layer
- dashboard in VS Code looks the same as the standalone report
- future report design changes automatically flow into the extension

## User Experience

## Command Palette Actions

Suggested commands:

- `Django Arch Check: Open Dashboard`
- `Django Arch Check: Refresh Dashboard`
- `Django Arch Check: Analyze Workspace`

Optional later:

- `Django Arch Check: Open Last HTML Report`
- `Django Arch Check: Export Dashboard`

## Webview Behavior

When the dashboard opens:

- the panel title should be `Django Architecture Dashboard`
- the panel should reuse the last opened panel if possible
- the extension should show a loading state while analysis runs
- if analysis succeeds, render the HTML
- if analysis fails, show a styled error state with actionable guidance

## Error UX

If the CLI is not installed:

- explain that `django-arch-check` was not found
- show the install command:

```bash
pip install django-arch-check
```

If no workspace is open:

- explain that the dashboard requires an open folder

If analysis exits with code `1`:

- this is not necessarily an error
- for this tool, exit code `1` may mean critical findings were found
- the extension should still show the generated report if the file exists

This is an important product detail.

## Phase 1 — Dashboard MVP

### Objectives

- run the CLI from VS Code
- generate `arch-report.html`
- render it in a webview

### Work Items

#### 1. Register Commands

Add commands in `package.json`:

- `djangoArchCheck.openDashboard`
- `djangoArchCheck.refreshDashboard`
- `djangoArchCheck.analyzeWorkspace`

#### 2. Workspace Discovery

Implement a helper that:

- uses the active workspace folder when available
- falls back to the first workspace folder
- warns clearly when no folder is open

#### 3. CLI Invocation

Create a small CLI service responsible for:

- locating the executable
- building arguments
- running the command
- collecting stdout/stderr
- returning exit code and paths

Preferred invocation:

```bash
django-arch-check analyze --format html <workspace>
```

#### 4. Handle Exit Code Semantics

Special-case this behavior:

- exit code `0`: clean or no critical findings
- exit code `1`: may still be a successful run with critical findings

Success criteria should be:

- `arch-report.html` exists and is readable

not just:

- process exited with `0`

#### 5. Read and Display HTML

Read the generated HTML file and feed it into a `WebviewPanel`.

Important:

- preserve CSS
- preserve inline JS
- preserve filters/theme switcher/interactions

#### 6. Add Loading and Error States

Before analysis completes, show a temporary HTML loading screen in the webview.

If something fails, show a styled failure view with:

- error summary
- stderr snippet
- install/setup hints

### Deliverables

- working command palette flow
- interactive dashboard opens inside VS Code

## Phase 2 — Reliability and Polish

### Objectives

- make the experience stable for real-world users

### Work Items

#### 1. Configurable CLI Path

Add VS Code settings:

```json
{
  "djangoArchCheck.cliPath": "django-arch-check",
  "djangoArchCheck.extraArgs": [],
  "djangoArchCheck.autoRefresh": false
}
```

This supports:

- global installs
- virtualenv paths
- pipx installs
- Windows-specific executable paths

#### 2. Preserve Panel State

If the user refreshes the dashboard:

- reuse the same Webview panel
- do not open duplicates unless explicitly requested

#### 3. Add Output Channel

Create a VS Code output channel:

- `Django Arch Check`

Use it to log:

- executed command
- resolved workspace
- stdout/stderr
- dashboard refresh status

#### 4. Improve Webview Safety

Because the report HTML contains inline scripts, decide how to load it safely.

Recommended approach:

- inject the HTML directly as `webview.html`
- add a CSP that allows the inline script pattern currently used by the report, or
- update the report generator later to emit CSP-friendly script blocks/nonces

This is one of the few places where some light coordination between the extension and Python package may be needed.

#### 5. Handle Relative Asset References

Right now the report is self-contained, which is ideal.

If future versions introduce separate assets:

- rewrite local URIs via `webview.asWebviewUri()`
- or keep the report self-contained by design

Recommendation:

Keep the report self-contained.

### Deliverables

- stable extension behavior
- settings support
- output channel
- improved diagnostics

## Phase 3 — Better Editor Integration

### Objectives

- make the dashboard feel more native to editor workflows

Support auto-refresh:

Do not auto-run by default. Architecture analysis is broader than linting and may be too noisy or expensive on every save.

- file

This is a strong phase-3 enhancement, but not required for the first release.

### Deliverables

## Phase 4 — Diagnostics and SARIF Mode

### Objectives

- extend beyond dashboard-only usage

### Why This Should Be Later

This phase is useful, but it is a different product surface.

The dashboard roadmap should not be blocked by:

- `DiagnosticCollection`
- SARIF parsing
- Problems panel synchronization

### Work Items

#### Option A — JSON-Based Diagnostics

Run:

```bash
django-arch-check analyze --format json <workspace>
```

Then map findings into VS Code diagnostics.

#### Option B — SARIF-Based Diagnostics

Run:

```bash
django-arch-check analyze --format sarif <workspace>
```

Then translate SARIF results into diagnostics.

### Recommendation

Use JSON first for direct extension integration.

Why:

- simpler payload
- already aligned with internal tool concepts
- easier to customize for editor UX

Use SARIF when targeting broader ecosystem integrations.

## Data Contract Considerations

## Current Strength

The package already produces:

- HTML report
- JSON output
- SARIF output

This is a strong foundation.

## Small Improvements That Would Help The Extension

These are optional, not blockers.

### 1. Stable Metadata Block In HTML

Add a small JSON payload inside the report HTML:

```html
<script id="arch-report-data" type="application/json">
  { ... }
</script>
```

That would let the extension:

- inspect report metadata
- add native toolbar actions later
- correlate cards to files/lines

### 2. Explicit Report Version Marker

Embed a report schema version:

```json
{
  "report_schema_version": 1
}
```

This helps the extension stay compatible as the HTML evolves.

### 3. Optional `--output` Flag For HTML

Current behavior writes `arch-report.html` to the project root.

An optional future improvement:

```bash
django-arch-check analyze --format html --output <path> <workspace>
```

Benefits:

- extension can write to a temp/cache directory
- avoids modifying the workspace if desired
- cleaner sandboxing for editor flows

This is a good future enhancement but not required for MVP.

## Cross-Platform Considerations

## Windows

Important because this project is actively used on Windows.

Watch for:

- `.exe` resolution
- virtualenv `Scripts` paths
- quoting workspace paths with spaces
- encoding issues in stdout/stderr

Recommendation:

- use Node `child_process.spawn` rather than building a raw shell command string
- pass args as an array
- read files as UTF-8

## macOS / Linux

Watch for:

- PATH differences between GUI-launched VS Code and shell sessions
- pyenv / poetry / pipx installs not being visible to the editor

This is why configurable `cliPath` matters.

---

## Security Considerations

## Webview Safety

The dashboard is generated locally and intended for the user’s own workspace, which lowers risk, but the extension should still be careful.

Principles:

- load only local generated content
- avoid enabling unnecessary Webview capabilities
- keep scripts local/self-contained
- prefer a minimal CSP where possible

## Shell Execution

Avoid:

- building shell commands as strings with unescaped workspace paths

Prefer:

- `spawn(executable, args, { cwd })`

## Risks

## Risk 1 — HTML/Webview Compatibility Drift

If the report HTML changes significantly, the extension could break if it relies on fragile assumptions.

Mitigation:

- treat the generated HTML as the primary render surface
- avoid scraping it unless necessary
- add a stable metadata block later if needed

## Risk 2 — CLI Discovery Friction

Users may install the package in environments VS Code cannot see.

Mitigation:

- configurable CLI path
- clear setup instructions
- output channel logging

## Risk 3 — Exit Code Misinterpretation

The extension might incorrectly treat critical findings as a run failure.

Mitigation:

- success = report file exists
- not success = only when report file is missing or CLI invocation truly fails

## Risk 4 — Scope Creep

Diagnostics, SARIF, code actions, and navigation can distract from shipping the dashboard.

Mitigation:

- ship dashboard MVP first
- keep diagnostics as phase 4

If the goal is to move fast, build exactly this first:

5. refresh command reruns analysis and replaces the HTML

That delivers real user value immediately and stays fully aligned with the current package.
